## Root cause — a dropped x86 carry in two `adc` sites

`THELOOP.INC` walks 192 depth steps front-to-back. `eax` is the fixed-point ray height (low 16 bits =
integer height `ax`, high 16 = fraction) and `ecx` is the fixed-point ray slope (negative — the ray
descends; it climbs toward 0 as the terrain rises). The fixed-point magnitudes are pre-`wflip`ped so the
integer part sits in the LOW word and the fraction in the HIGH word, hence the recurring `add …; adc …,0`
pattern (carry from the fractional low add ripples into the integer high word).

Two places use x86 **`adc` (add-with-carry)** where the carry-in is the carry-OUT of the immediately
preceding `add`. The port had emulated both as a plain `add32`, silently dropping that carry:

1. `_@seek2a` — the **2-row hit** path. THELOOP.INC:

   ```asm
   _@seek2a:
     sub bp,160
     add ecx,0A000000h      ; CF_a = carry-out
   _@seek1a:
     adc ecx,0A000000h      ; ecx += 2560 + CF_a   <-- carry-in MUST be threaded
     adc cx,0
     sub bp,160
   ```

   `L_2560 = wflip(2560) = 0x0A000000`. As `ecx` climbs past `0xF6000000` the first `add` carries out of
   32 bits, so `CF_a = 1`. The old port did `add32(ecx, L_2560)` then `add32(ecx, L_2560)` (no carry-in),
   losing that +1 in the slope's fraction.

2. The **sina2 `_@seeko`** exit (`j >= 64`, the far-LOD half that marches two depth steps at once).
   THELOOP.INC:

   ```asm
   _@seeko64:
     add eax,ecx
     adc eax,ecx            ; eax += ecx + CF   <-- carry-in MUST be threaded
     adc ax,-1
   ```

   The old port did `add32(eax, ecx)` then, for `sina===2`, another plain `add32(eax, ecx)`.

Either dropped carry leaves the slope/height one fixed-point unit too shallow for that column. The ray then
descends a hair too slowly, never clears the terrain, and the per-hit fill loop keeps writing rows upward
(`mov es:[bp],dl; sub bp,160`) — the original has no upper bound; it relies entirely on `ax >= bx`
terminating it. The result is one column filled dozens of extra rows: a tall 1px-wide spike. The original
cannot produce this because adjacent columns sample neighbouring heightfield cells, so the terrain top
varies only gently across the screen.

Bug 1 alone shifted one repro column's terrain top from row 78 to row 28; bug 2 added the rest, landing on
the observed row-24 spike.

## Fix

`packages/parts/src/comanche/raster.ts`:

- Added an `adc32(a, b, cf)` helper (`a + b + carry-in`, with its own carry-out) modelling `adc r32,src`.
- `_@seek2a` (2-row path): `[ecx, cfA] = add32(ecx, L_2560); [ecx, cf2] = adc32(ecx, L_2560, cfA);` then
  `adc cx,0`.
- Both `_@seeko` exits (the inline no-hit skip and the post-hit exit): the sina2 second add is now
  `adc32(eax, ecx, cf)` instead of `add32(eax, ecx)`.

Nothing else changed — `add32`, `adcLow16`, the `wflip` constants, the `bx` heightfield read
(`heightX[(s>>1) & (WAVE_FIELD_WORDS-1)]`, 16-bit-wrapped add of the two fields + `zwave[j]-240`), the
`sina` 1->2 switch at depth 64, and the shade index are untouched.

## Verification — independent oracle

The shipped `raster-columns.json` oracle had been captured from the (buggy) `rasterColumn` itself, so it
just locked the bug in. It was **re-derived independently**: a from-scratch Python emulation of the actual
shipped `THELOOP.INC` opcodes (`add`/`adc`/`cmp`/`jge`, 16-bit `ax`/`bx` signed compare, the `wflip`
fixed-point) run against the real `W1DTA.BIN` / `W2DTA.BIN`. That reference produces NO spikes (terrain
tops cluster within a few rows across all 160 columns) for every camera frame sampled. The fixed
`rasterColumn` matches that reference **byte-for-byte** across 2240 columns (160 columns × 14 frames
spanning the intro rise and steady-state). The new oracle holds the 5 original inputs plus 3 frame-60
columns that previously spiked (`[6104,11712,2,78]` etc.).

A regression test (`raster.test.ts`, "the field is smooth rolling hills — no spurious tall single-column
spikes") rasterises a real frame and asserts no column's terrain top juts more than 8 rows above either
neighbour (a spike jutted tens).
